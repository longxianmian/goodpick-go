# GoodPick Go Staff OA / 店员工作台 功能规格说明（给 Replit AI）

> 目标：在现有 GoodPick Go H5 多语言系统基础上，增加**店员工作用的 3 个页面**，供“工作用 OA（Staff OA）”使用。工作 OA 只包含店员相关功能，不展示用户优惠券。

---

## 一、入口与路由约定

### 1.1 复用现有基础设施

* 继续复用现有的：

  * LINE Login Channel
  * LIFF 应用（同一个 LIFF ID）
  * Replit H5 域名（当前项目的公共域名）
* 不新建新的 Channel / LIFF，只是增加新的入口 view 和页面。

### 1.2 Staff OA 的富菜单（仅说明链接约定）

在工作 OA（Staff OA）的 Rich Menu 中，会配置 3 个按钮：

1. **核销入口按钮**：链接到 `?view=staff-redeem`
2. **活动说明 / 培训入口按钮**：链接到 `?view=staff-campaign`
3. **个人统计入口按钮**：链接到 `?view=staff-stats`

具体 LIFF URL 形式：

```text
https://liff.line.me/{LIFF_ID}?view=staff-redeem
https://liff.line.me/{LIFF_ID}?view=staff-campaign
https://liff.line.me/{LIFF_ID}?view=staff-stats
```

### 1.3 前端入口 view 分流

在现有根据 `view` 参数跳转的逻辑上，增加 3 个分支：

```ts
const params = new URLSearchParams(window.location.search);
const view = params.get("view");

if (view === "staff-redeem") {
  navigate("/staff/redeem", { replace: true });
} else if (view === "staff-campaign") {
  navigate("/staff/campaign", { replace: true });
} else if (view === "staff-stats") {
  navigate("/staff/stats", { replace: true });
}
// 其他 view（campaign / my-coupons 等）沿用现有逻辑
```

在 React Router 中增加 3 条路由：

```tsx
<Routes>
  {/* 已有路由保留 */}
  <Route path="/campaign/:id" element={<CampaignDetail />} />
  <Route path="/my-coupons" element={<MyCoupons />} />

  {/* 新增：店员工作台路由 */}
  <Route path="/staff/redeem" element={<StaffRedeemPage />} />
  <Route path="/staff/campaign" element={<StaffCampaignPage />} />
  <Route path="/staff/stats" element={<StaffStatsPage />} />
</Routes>
```

## 二、店员页面设计（前端功能）

> 要求：
>
> * 所有页面为移动端优先布局，样式延续现有 H5 风格。
> * 所有文案使用现有多语言机制（i18n），**不在组件中硬编码中文 / 英文 / 泰文文案**；请使用文案 key（例如 `t('staff.redeem.title')`）。

### 2.1 `/staff/redeem` —— 店员核销页面

这是 Staff OA 的核心功能页面，对应 Rich Menu 中的“核销入口”按钮。

#### 2.1.1 登录与权限

> ⚠️ 本小节必须**结合现有“门店管理 → 核销授权”模块**来实现，不能再假设“谁拿到 OA 就是店员”。

* 进入 `/staff/redeem` 时，先执行和用户端相同的登录逻辑：

  * LINE 登录 + JWT 认证
  * 若未登录，按现有统一逻辑触发 LINE 登录流程
* 登录成功后，后端需要基于现有的“门店核销授权”数据做**店员身份校验**：

  * 在核销授权表中，根据当前用户的 `line_user_id`（或等价字段）查找**已激活的员工授权记录**；
  * 只有存在“已授权 & 已激活”的员工记录时，才允许访问 Staff OA 页面和 `/api/staff/**` 接口；
  * 若找不到授权记录，接口统一返回“无核销权限”（错误码可以复用现有规范），前端展示友好的提示，例如“请联系门店管理员开通核销权限”。
* 员工授权的创建与二维码发放逻辑沿用现有“门店管理 → 核销授权”弹窗：

  * 后台为门店添加员工信息（姓名、员工编号、电话等）；
  * 系统生成授权链接 + 二维码；
  * 员工用自己的 LINE 账号扫码授权链接，完成账号绑定和激活；
  * 绑定逻辑由后端负责，将该员工记录与当前 `line_user_id` 关联，并将状态从“待绑定/待激活”更新为“已激活”。

#### 2.1.2 页面结构（MVP）

页面主要包含三个区域：

1. 顶部：当前登录用户信息

   * 头像、昵称（复用现有 user 信息）
2. 中部：**8 位手动核销码输入 + 核销按钮**

   * 单个文本输入框，用于输入“手动核销码”；
   * 手动核销码统一约定为 **8 位阿拉伯数字**（`0-9`），例如 `08342579`：

     * 前端需要做基础校验：只能输入数字；长度不足 8 位时禁用“核销”按钮；长度超过 8 位时截断或提示错误；
     * 占位文案和错误提示通过 i18n 获取，不写死文案；
   * 核销按钮：点击后调用后台核销接口；
3. 底部：最近核销记录列表（最多 5 条）

   * 每条记录显示：

     * 核销时间
     * 活动标题
     * 券码后 4 位（这里的“券码”指 8 位手动核销码的后 4 位，例如 `***2579`）。

> 说明：用户侧 H5 的“我的优惠券”页面中，也需要展示同一个 8 位核销码，店员在本页面输入的就是这一串数字；长券码、二维码等可以内部使用，但**人工输入统一以 8 位数字核销码为准**。

#### 2.1.3 行为逻辑

* 点击“核销”按钮时：

  * 调用后端接口：`POST /api/staff/coupons/redeem`
  * 请求体示例：

    ```json
    ```

{ "manualCode": "08342579" }

````

- `manualCode`：必填，字符串形式的 8 位数字手动核销码；
- 前端已做基础格式校验，但后端仍需再次校验：
  - 必须为长度 8 的数字字符串；
  - 在数据库中查找对应优惠券记录时，以此字段进行匹配。

  - 正常流程：
    - 校验券码合法、未使用、未过期
    - 更新状态为已使用
    - 记录当前登录用户为核销人
    - 返回优惠券与活动的关键信息
    - 前端展示成功提示 + 当前券详情（活动名、优惠金额、有效期等）
    - 刷新最近核销记录列表
  - 异常情况：
    - 券码不存在
    - 已使用
    - 已过期
    - 其他业务错误
  - 前端需要根据错误信息展示友好的多语言错误提示。

### 2.2 `/staff/campaign` —— 当前活动说明 / 培训页面

此页面用于向店员展示当前参与的活动信息、执行要点和培训内容，对应 Rich Menu 的“活动说明 / 培训入口”按钮。

#### 2.2.1 登录

- 同样依赖已有的 LINE 登录 + JWT 认证

#### 2.2.2 功能需求

1. 展示当前进行中的活动列表：
   - MVP 可以先展示所有 `active` 活动
   - 若后续需要按门店过滤，再在后端扩展
2. 每个活动显示：
   - 活动标题（根据当前语言，从多语言字段中选择对应版本）
   - 活动时间（开始 / 结束）
   - 活动简介（短描述）
3. 可展开查看详情，详情包含：
   - 更完整的活动说明
   - 执行步骤（前端可以先用固定模板文案占位，后续可增加字段映射）
   - 店内话术 / 注意事项 / 考核信息（同样可先用占位文案）

#### 2.2.3 API 建议

新增接口：`GET /api/staff/campaigns`

- 返回当前系统中 `active` 的活动列表
- 返回字段示例：

```json
{
  "success": true,
  "data": [
    {
      "id": 1,
      "title": "儿童智能手表抵用现金券",
      "description": "活动说明...",
      "startAt": "2025-11-01T00:00:00Z",
      "endAt": "2025-11-30T23:59:59Z"
    }
  ]
}
````

> 说明：`title` / `description` 字段由后端根据当前语言（Accept-Language 或用户语言偏好）选择对应多语言字段后返回，前端直接展示。

### 2.3 `/staff/stats` —— 店员个人核销统计页面

此页面用于展示当前登录店员个人的核销数据和近期核销记录，对应 Rich Menu 的“个人统计入口”按钮。

#### 2.3.1 登录

* 需要登录（JWT），与前述页面一致

#### 2.3.2 功能需求

页面包含三部分信息：

1. 汇总统计

   * 今日核销量
   * 本周核销量
   * 本月核销量
2. 按活动统计

   * 每个活动：活动标题 + 当前店员在该活动下的核销数量
3. 最近核销记录列表

   * 最近 N 条记录（例如 10 条）
   * 每条记录包含：核销时间、活动标题、券码后 4 位

#### 2.3.3 API 设计

新增两个接口：

1. `GET /api/staff/summary`

   ```json
   {
     "success": true,
     "data": {
       "today": 3,
       "thisWeek": 12,
       "thisMonth": 45,
       "byCampaign": [
         { "campaignId": 1, "title": "儿童智能手表", "count": 30 },
         { "campaignId": 2, "title": "耳机活动", "count": 15 }
       ]
     }
   }
   ```

2. `GET /api/staff/recent-redemptions?limit=10`

   ```json
   {
     "success": true,
     "data": [
       {
         "couponId": 123,
         "codeLast4": "8F2C",
         "campaignTitle": "儿童智能手表",
         "redeemedAt": "2025-11-05T11:30:00Z"
       }
     ]
   }
   ```

> 说明：`title` / `campaignTitle` 等同样由后端根据当前语言选择正确的多语言字段返回。

## 三、后端改造与权限

### 3.1 认证与权限原则

* 所有 `/api/staff/**` 接口，都必须在现有 JWT 认证中间件之后调用：

  * 从 JWT 中获取当前用户 ID 及其 `line_user_id`（或等价字段）；
* 在通过 JWT 校验后，**还需要做店员授权校验**：

  * 基于“门店管理 → 核销授权”模块中的员工授权表，按 `line_user_id` 查找“已激活”的员工记录；
  * 找不到记录或状态不是“已激活”时，直接返回无权限错误（例如 `success: false, message: 'NO_STAFF_PERMISSION'`），前端按多语言提示“您还没有核销权限，请联系门店管理员”；
  * 找到记录时，将该员工记录（包含所属门店信息）注入到请求上下文，供后续接口使用，例如记录 `redeemed_store_id`、统计按门店/员工维度核销量等。
* 这样实现后：

  * 普通用户即便知道 Staff OA 的 URL，也无法通过授权校验；
  * 只有在“门店管理 → 核销授权”中被创建并完成扫码绑定的员工，才能正常使用核销 & 统计功能。

### 3.2 与现有“门店核销授权”模块的对接

> 这一节只描述对接约束，不强制具体表名；请在实现时复用现有门店/员工授权的数据结构，避免再造一个新的 staff 表。

* 现有后台中已经有“门店管理 → 核销授权”弹窗，包含：员工姓名、员工编号、电话、状态、授权二维码与授权链接等字段；
* 在此基础上需要补充/确认：

  * 每条员工授权记录具备唯一 ID；
  * 具备状态字段（例如：待激活 / 已激活 / 已禁用）；
  * 具备一个用于绑定 LINE 账号的标识字段（例如 `line_user_id`）。
* 扫码授权链接时的期望行为：

  1. 通过授权链接中的 token 或员工 ID 找到对应的授权记录；
  2. 触发 LINE 登录（如果尚未登录）；
  3. 登录成功后，将当前用户的 `line_user_id` 写入该授权记录，并将状态调整为“已激活”；
  4. 可以在前端展示绑定成功页面，并提示去 Staff OA 菜单使用核销功能。
* Staff OA 的 3 个页面（`/staff/redeem`、`/staff/campaign`、`/staff/stats`）以及所有 `/api/staff/**` 接口，都必须基于上述员工授权记录进行权限判断，不再依赖“谁拿到 OA 就默认是店员”的假设。

### 3.3 新增接口列表汇总

1. 核销接口：

   * `POST /api/staff/coupons/redeem`
   * 入参：

     * `manualCode`：字符串，**8 位数字的手动核销码**，例如 `08342579`；
   * 约束与设计：

     * 每张优惠券必须拥有一个唯一的 8 位手动核销码字段（可以命名为 `manual_code` 或 `short_code` 等）；
     * 该字段在券生成时即写入，并展示在用户端“我的优惠券 / 券详情”中，供店员人工输入；
     * 手动核销接口只通过此字段查找券记录，不依赖长券码；
   * 行为：

     * 按 `manualCode` 查询对应券码记录；
     * 校验状态 / 过期时间；
     * 更新为已使用，写入 `used_at`、`redeemed_store_id`（如需要）、`redeemed_by_user_id`（可在 coupons 表新增该字段）；
     * 返回当前券及其活动的关键信息。

2. 活动列表接口：

   * `GET /api/staff/campaigns`
   * 行为：

     * 查询 `active` 活动
     * 根据用户语言返回对应语言的标题 / 描述

3. 统计与记录接口：

   * `GET /api/staff/summary`
   * `GET /api/staff/recent-redemptions?limit=10`

所有接口返回结构与当前项目的 `success / data / message` 约定保持一致。

   * `/staff/redeem` 页面

   * `/staff/stats` 页面

3. **第三优先级**：

   * `/staff/campaign` 页面
   * `GET /api/staff/campaigns`
     → 把活动说明和简单培训内容放进去。

## 五、多语言实现要求（重要）

GoodPick Go 系统的多语言由两部分组成：**系统 UI 文案** 和 **业务动态内容**。本次 Staff OA 与 Dashboard 开发必须同时兼容这两部分的既有方案，而不是临时增加语言按钮或在代码中硬编码多种语言。

### 5.1 系统 UI 文案（固定文案）

指菜单标题、按钮文字、表头、错误提示、说明性文字等**相对固定**的界面文案，例如：

* “核销入口”“活动说明”“个人统计”“数据面板”
* “立即核销”“今日核销量”“本月核销量”“最近核销记录”等

实现要求：

1. 统一使用现有的 i18n 方案（如 `t('staff.redeem.title')`），**不在组件中直接写任何语言的文字**。
2. 新增页面（`/staff/redeem`、`/staff/campaign`、`/staff/stats`、`/admin/dashboard`）需要在 i18n 资源文件中新增对应 key：

   * 例如：`staff.redeem.title`、`staff.redeem.manualCodePlaceholder`、`staff.stats.today`、`dashboard.campaigns.title` 等；
3. 切换系统语言时，Staff OA 与 Dashboard 页面应与现有 H5 一致，自动切换 UI 文案，不单独再做语言选择器。

### 5.2 业务动态内容（活动标题 / 规则 / 培训文案等）

指会随活动和品牌变化而变化的文本，例如：

* 活动标题、活动短描述、活动详细规则；
* 门店培训说明、话术、执行注意事项；
* 品牌方提供的活动说明长文本等。

这些内容**不能**全部写死在前端 i18n，而是按以下流程处理：

1. **创建或编辑活动时**，运营人员在后台只需要维护一个“源语言版本”（`*_source_lang` + `*_source`），例如先用中文或泰文填写：

   * `title_source_lang` + `title_source`
   * `description_source_lang` + `description_source`
   * 可选：`rules_source`、`staff_training_source` 等字段（根据实际 schema 扩展）。
2. 保存活动时，后端调用现有的 **OpenAI 翻译服务**（`translationService`）：

   * 将源语言内容翻译为系统支持的其他语言（目前至少中/英/泰三种）；
   * 将结果分别写入结构化字段：例如 `title_zh`、`title_en`、`title_th`；`description_zh`、`description_en`、`description_th`；如有规则/培训字段，同样按语言拆分存储；
   * 翻译结果只在**保存/更新活动**时调用一次，不在每次用户访问时实时翻译，避免成本和延迟。
3. 对于**体量较大或富文本的内容**（例如图文培训手册）：

   * 可以将原文及翻译后的版本作为 JSON 或 Markdown 存入数据库或 OSS 对象存储中，并在活动记录中保存引用路径；
   * 前端根据当前语言从接口拿到相应版本的文本或链接，直接渲染，不再调用 OpenAI。
4. 当管理员在后台修改源语言内容时：

   * 需要重新调用翻译服务覆盖对应语言字段，保证多语言内容同步。

### 5.3 接口返回与前端使用规范

1. 所有返回“动态内容”的接口（包括用户端与 Staff OA、Dashboard，例如 `/api/campaigns/:id`、`/api/staff/campaigns`、`/api/admin/dashboard/**`），都应由后端根据当前请求语言或用户语言偏好，从 `*_zh` / `*_en` / `*_th` 等字段中选取一个最合适的版本，再返回给前端：

   * 前端不关心内部是如何翻译的，只消费一个 `title` / `description` / `rules` 字段即可；
   * 若某语言缺失，可回退到源语言或英文，具体策略由后端统一处理。
2. Staff OA 页面中出现的所有“活动相关文案”（标题、描述、培训说明等），以及 Dashboard 中的活动名称等，都必须使用上述“动态内容”字段，**不要在前端写死多语言文本，也不要在前端再次调用 OpenAI**。
3. 多语言切换时：

   * UI 固定文案通过 i18n 自动切换；
   * 动态内容由接口根据语言切换返回对应版本；
   * 前端不需要做额外逻辑，只需在切换语言后重新请求数据。

> 总结：
>
> * **UI 层文案**：全部走 i18n key，不写死任何语言。
> * **业务动态内容**：由后台在写入/更新时通过 OpenAI 翻译并结构化存储，接口按当前语言返回对应字段。前端只负责展示。

## 六、运营后台数据面板（Admin Dashboard）规范

> 目标：让系统管理员可以在「数据面板」中，按月份查看 GoodPick Go 整体运营效果，并从**活动维度 / 品牌维度 / 门店维度**三个角度进行统计分析。该面板仅对后台管理员开放，不对普通用户和店员开放。

### 6.1 页面位置与基础交互

* 后台路由：`/admin/dashboard`（当前菜单「数据面板」页）。
* 只允许管理员 JWT 登录后访问，沿用现有 admin 权限体系。
* 页面顶部提供：

  * **月份选择器**：

    * 格式示例：`2025-11`，只选年月（不选具体日），默认当前月份；
    * 切换月份后，重新请求后台数据；
  * （可选）品牌过滤器：用于在门店维度中按品牌筛选。
* 页面主体建议分为 3 个卡片区域：

  1. 活动维度统计
  2. 品牌维度统计
  3. 门店维度统计

> 注意：所有金额、数量等统计逻辑统一以 **券领取记录（coupons）与活动（campaigns）** 为基础进行聚合，避免在前端自行拼装统计。所有聚合工作在后端完成。

### 6.2 统计口径约定（统一规则）

为避免歧义，所有维度统计共用以下口径：

* 统计周期：

  * 由前端传入 `month` 参数，例如 `2025-11`；
  * 后端将其换算为：

    * `start = 2025-11-01 00:00:00`
    * `end   = 2025-11-30 23:59:59`（按实际月份天数计算）；
* **参与活动范围**：

  * 活动开始时间与结束时间与该月份有重叠即可算在内（例如 10 月开始、12 月结束的活动，在 11 月依然算有效活动）。
* **领券数量（claimedCount）**：

  * `coupons.issued_at` 在本月时间范围内的券数量；
  * 状态不限（未使用 / 已使用 / 已过期都算领取过）。
* **核销数量（redeemedCount）**：

  * `coupons.status = 'used'`，且 `coupons.used_at` 在本月时间范围内的数量；
* **参与门店数量（storeCount）**：

  * 在本月内，至少有一张券被领取或核销的门店数量（基于 `redeemed_store_id` 和活动关联门店 `campaign_stores` 共同统计，可按实现情况选择更简单的一种：例如只按 `campaign_stores` 统计该活动挂了多少门店）。
* **转化率（conversionRate）**：

  * 若 `claimedCount > 0`，`conversionRate = redeemedCount / claimedCount`，以百分比形式返回；
  * 若 `claimedCount = 0`，转化率为 0 或 `null`，由前端处理展示。

### 6.3 活动维度统计（By Campaign）

#### 6.3.1 数据展示需求

在「活动维度」卡片中，以表格形式展示本月所有相关活动，每行包含：

* 活动 ID
* 活动名称（按当前后台语言返回对应多语言标题）
* 所属品牌（可选，若活动本身有品牌字段，可以展示；否则可用主要门店品牌占位）
* 活动期间（开始日期 - 结束日期）
* 参与门店数量 `storeCount`
* 本月领券数量 `claimedCount`
* 本月核销数量 `redeemedCount`
* 本月转化率 `conversionRate`

表格支持：

* 按领券数 / 核销数 / 转化率排序；
* 点击活动行可以跳转到该活动详情页（现有活动管理页），便于继续调整配置。

#### 6.3.2 后端接口建议

新增接口：

```http
GET /api/admin/dashboard/campaigns?month=2025-11
```

返回示例：

```json
{
  "success": true,
  "data": [
    {
      "campaignId": 1,
      "title": "儿童智能手表抵用现金券",
      "brand": "BaNANA",          
      "startAt": "2025-11-01T00:00:00Z",
      "endAt": "2025-11-30T23:59:59Z",
      "storeCount": 25,
      "claimedCount": 350,
      "redeemedCount": 210,
      "conversionRate": 0.6
    }
  ]
}
```

> 说明：`title` / `brand` 字段仍由后端根据当前管理员语言返回正确的多语言文本；前端只负责展示。

### 6.4 品牌维度统计（By Brand）

#### 6.4.1 数据展示需求

品牌维度用于从更高层级查看整体合作品牌表现，例如：BaNANA、7-Eleven、KFC 等。

每个品牌一行，字段包括：

* 品牌名称 `brand`
* 本月参与活动数量 `campaignCount`
* 本月参与门店数量 `storeCount`
* 本月总领券数量 `claimedCount`
* 本月总核销数量 `redeemedCount`
* 本月平均转化率 `conversionRate`（可以用所有活动汇总后重新按公式计算，而不是简单平均）

前端可以用：

* 表格 + 条形图（可选）展示对比；
* 点击品牌行时，在右侧或下方联动展示该品牌下的活动列表（复用 6.3 的数据，前端筛选即可）。

#### 6.4.2 后端接口建议

```http
GET /api/admin/dashboard/brands?month=2025-11
```

返回示例：

```json
{
  "success": true,
  "data": [
    {
      "brand": "BaNANA",
      "campaignCount": 2,
      "storeCount": 40,
      "claimedCount": 500,
      "redeemedCount": 320,
      "conversionRate": 0.64
    },
    {
      "brand": "7-Eleven",
      "campaignCount": 1,
      "storeCount": 120,
      "claimedCount": 800,
      "redeemedCount": 480,
      "conversionRate": 0.6
    }
  ]
}
```

> 品牌字段可以直接复用 `stores.brand` 或 `campaigns.brand`（根据实际数据结构选择其中之一），后端在聚合时只需保持统一口径即可。

### 6.5 门店维度统计（By Store）

#### 6.5.1 数据展示需求

门店维度用于精细化查看每家门店的执行情况。建议使用带分页的表格：

* 门店名称
* 品牌
* 城市（可选）
* 本月参与活动数量 `campaignCount`
* 本月领券数量 `claimedCount`
* 本月核销数量 `redeemedCount`
* 本月转化率 `conversionRate`

支持筛选与排序：

* 按品牌筛选（顶部品牌下拉，与 6.4 共享）；
* 按领券数 / 核销数 / 转化率排序；
* 按门店名称搜索。

#### 6.5.2 后端接口建议

```http
GET /api/admin/dashboard/stores?month=2025-11&page=1&limit=20&brand=BaNANA
```

返回示例：

```json
{
  "success": true,
  "data": [
    {
      "storeId": 101,
      "name": "7-Eleven สาขา Booth อินทราแสควร์",
      "brand": "7-Eleven",
      "city": "Bangkok",
      "campaignCount": 1,
      "claimedCount": 35,
      "redeemedCount": 22,
      "conversionRate": 0.63
    }
  ],
  "pagination": {
    "page": 1,
    "limit": 20,
    "total": 200,
    "totalPages": 10
  }
}
```

> 说明：门店维度依赖 `stores`、`campaign_stores`、`coupons` 三张表的聚合关系，具体实现可按当前数据库 schema 做调整，但对前端返回结构保持一致。

### 6.6 总览卡片（可选增强）

为方便运营快速掌握整体情况，可以在页面顶部增加一个「本月概览」卡片，展示：

* 本月活动总数
* 本月合作品牌数
* 本月参与门店数
* 本月总领券数
* 本月总核销数
* 本月整体转化率

接口可以与上述某个接口复用（例如在 `GET /api/admin/dashboard/campaigns` 中额外返回一个 `summary` 字段），或单独提供：

```http
GET /api/admin/dashboard/summary?month=2025-11
```

返回整体汇总数据即可。

### 6.7 多语言与前端实现要求

* 所有标题、表头、提示文案继续使用现有的全局 i18n 方案获取，不在组件中写死中文 / 英文 / 泰文文案；
* 活动名称、品牌名称、门店名称等字段：

  * 如已存多语言版本，由后端根据当前管理员语言选择合适字段返回；
  * 如仅存一种语言（例如门店名称为泰文），则直接原样返回即可；
* 前端数据面板仅负责可视化展示与交互（筛选、排序、分页），不在前端自行做复杂聚合运算，避免与后端口径不一致；
* 所有 Dashboard 相关接口路径统一归类到 `/api/admin/dashboard/**`，并沿用当前 `success / data / message` 的响应结构规范。
